## 2. 알고리즘 진화 과정

### 2.1 전체 흐름: C1 → C14 → C25

개발은 크게 **3단계**로 진행되었다.

#### Phase 1: 기본 전략 수립 및 초기 최적화 (C1~C13)

- **C1 (Baseline)**: 고정 금액 매수/매도. V4 스코어 기반 단순 진입/퇴장.
- **C2~C5**: 고정 금액 → **현금 비율 기반** 전략으로 전환. 복리 효과 확보.
- **C6~C9**: ER(Efficiency Ratio) + ATR(Average True Range) **가격 필터** 도입. 노이즈 신호 제거.
- **C10~C13**: 매수/매도 비율 분리, 강한 신호/일반 신호 구분 체계 도입, 월 $100 DCA(Dollar Cost Averaging) 추가.
- **테스트 방법**: 7개 시장 구간 × 40개 종목 크로스 백테스트.

#### Phase 2: C14 확정 및 심층 분석

- **C14 = PRICE_FILTER_RELAX + LATE_SELL_BLOCK**:
  - ATR quantile 66 → 55로 완화하여 더 많은 신호 통과
  - 20일 롤링 고점 대비 5% 이상 하락 시 매도 차단 (늦은 매도 방지)
- **C14 결과**: 연환산 11.45%, MDD -48.8%
- **C14 심층 분석** (analyze_c14.py, 997줄):
  - 10개 분석 영역에서 종합 평가
  - **치명적 문제 발견**: STRONG_BUY_TH = SIGNAL_THRESHOLD = 0.15 → 모든 매수가 "강한 매수"로 분류
  - 8가지 개선 권고안 도출

#### Phase 3: C25 최적화 백테스트 및 확정

- **20개 구성 테스트**: C14 베이스 + 개별 8개(I1~I8) + 조합 11개(C15~C25)
- **C25가 최종 1위**로 채택
- 전체 프로덕션 시스템 코드에 C25 알고리즘 반영 완료

### 2.2 Phase 3 개선안 상세

C14 심층 분석에서 도출된 8가지 개선안:

| ID | 개선안 | 변경 내용 | 목적 |
|----|--------|----------|------|

## 4. C25 알고리즘 상세

### 4.1 V4_wP 지표 계산

```
V4_Score = (0.3 × S_Force + 0.4 × S_Div + 0.3 × S_Conc) × Activity_Multiplier
```

| 서브지표 | 가중치 | 설명 |
|----------|--------|------|
| S_Force (거래량 힘) | 30% | 매수/매도 압력 비율 |
| S_Divergence (괴리) | 40% | 가격-거래량 괴리 측정 |
| S_Concordance (일치) | 30% | 가격-거래량 방향 일치도 |
| Activity Multiplier | ×1 | 거래 활발도에 따른 배율 조정 |

### 4.2 신호 탐지 파이프라인

```
[OHLCV 데이터]
    ↓
[V4 Score 계산] (window=20)
    ↓
[피크/바텀 감지] (threshold=0.15, cooldown=5)
    ↓
[가격 필터] (ER quantile=66, ATR quantile=55)
    ↓
[LATE_SELL_BLOCK] (20일 고점 대비 5% 하락 시 매도 차단)
    ↓
[신호 강도 분류] (C25: 강매수 TH=0.25)
    ↓
[매수/매도 비율 결정]
```

### 4.3 C25 파라미터 전체

```json
{
    "v4_window": 20,
    "signal_threshold": 0.15,
    "cooldown": 5,
    "er_quantile": 66,
    "atr_quantile": 55,
    "lookback": 252,
    "data_years": 3,
    "strong_buy_th": 0.25,
    "strong_sell_th": -0.25,
    "buy_normal_pct": 0.40,
    "buy_strong_pct": 0.60,
    "sell_normal_pct": 0.05,
    "sell_strong_pct": 0.10,
    "late_sell_drop_th": 0.05
}
```

### 4.4 신호 분류 및 행동 규칙

| 신호 | 조건 | 행동 |
|------|------|------|
| **STRONG BUY** | V4 bottom, \|peak_val\| ≥ 0.25 | 가용 현금의 **60%** 매수 |
| **BUY (일반)** | V4 bottom, \|peak_val\| < 0.25 | 가용 현금의 **40%** 매수 |
| **STRONG SELL** | V4 top, peak_val ≤ -0.25 | 보유 주식의 **10%** 매도 |
| **SELL (일반)** | V4 top, peak_val > -0.25 | 보유 주식의 **5%** 매도 |
| **BLOCKED SELL** | 20일 고점 대비 5%+ 하락 상태 | **매도 차단** (이미 떨어진 뒤 팔지 않음) |

### 4.5 LATE_SELL_BLOCK 로직

```python
rolling_high_20 = Close.rolling(20, min_periods=1).max()
drop_pct = (rolling_high_20 - current_price) / rolling_high_20

if drop_pct > 0.05:  # 5% 이상 하락
    → 매도 신호 차단 (이미 늦은 매도)
```

**효과**: 급락 이후 바닥에서의 투매를 방지. 특히 Lehman, Covid 기간에 효과적.

## 6. 핵심 모듈 상세

### 6.1 `core/indicators.py` — V4 지표 + C25 신호 분류

**주요 함수:**

| 함수 | 역할 |
|------|------|
| `fetch_data(ticker, years)` | yfinance로 최근 N년 OHLCV 다운로드 |
| `analyze_ticker(ticker, df, params)` | V4 스코어 → 신호 감지 → 가격 필터 → LATE_SELL_BLOCK |
| `classify_signal(ev, params)` | **C25 핵심**: 신호 강도 분류 (STRONG/NORMAL × BUY/SELL) |
| `get_latest_score_data(df, subind, n_days)` | 최근 N일 스코어를 dict로 변환 (DB 저장용) |

**`analyze_ticker()` 반환값:**
```python
{
    'score': Series,          # V4 스코어
    'subindicators': DataFrame,  # s_force, s_div, s_conc, act, score
    'all_events': list,       # 필터 전 전체 이벤트
    'filtered_events': list,  # 가격필터 + LATE_SELL_BLOCK 적용 후
    'blocked_sells': list,    # 차단된 매도 이벤트
    'price_filter_fn': callable,
}
```

**`classify_signal()` 반환값:**
```python
{
    'is_strong': bool,        # 강한 신호 여부
    'label': str,             # 'STRONG BUY' / 'BUY' / 'STRONG SELL' / 'SELL'
    'action_pct': float,      # 매수/매도 비율 (0.40/0.60/0.05/0.10)
}
```

### 6.2 `core/scanner.py` — 일일 스캔 엔진

**`run_scan(alert_fn, commentary_fn, dry_run)` 흐름:**

1. watchlist.json에서 종목 + 파라미터 로드
2. DB 초기화 (dry_run이 아닌 경우)
3. 각 종목에 대해:
   - `fetch_data()` → `analyze_ticker()` → `get_latest_score_data()`
   - 일별 스코어 DB 저장 (`upsert_daily_scores`)
   - 차단된 매도 기록

## 9. C14에서 발견된 문제와 C25의 해결

### 9.1 STRONG_BUY_TH = SIGNAL_THRESHOLD 문제 (가장 중요)

**문제:**
```
C14: STRONG_BUY_TH = 0.15 = SIGNAL_THRESHOLD
→ 신호가 발생하려면 |peak_val| >= 0.15 필요
→ 모든 매수 신호가 자동으로 "강한 매수"로 분류
→ 강한/일반 구분이 사실상 무의미
→ 항상 50% 현금 투입 (과도한 집중)
```

**C25 해결:**
```
C25: STRONG_BUY_TH = 0.25 > SIGNAL_THRESHOLD = 0.15
→ |peak_val| 0.15~0.24: 일반 매수 (40% 현금)
→ |peak_val| 0.25+: 강한 매수 (60% 현금)
→ 의미 있는 2단계 분류 복원
→ 약한 신호에 적정 비율, 강한 신호에 집중 투자
```

### 9.2 워치리스트 문제

**문제:**
```
PGY: 전기간 -8.2% 연환산 (지속 하락)
BA: 전기간 -3.1% 연환산 (737 MAX 이슈)
INTC: 전기간 -1.5% 연환산 (시장 점유율 하락)
→ 3 종목이 전체 포트폴리오 성과를 끌어내림
```

**C25 해결:**
```
PGY, BA, INTC 워치리스트에서 제거
→ 40종목 → 37종목
→ 단독으로 +0.81%p 효과 (가장 큰 개선)
```
